🔧 ARCHITECTURE TECHNIQUE

Cycle de fonctionnement :
┌─────────────────────────────────────┐
│ initialize() au démarrage            │
├─────────────────────────────────────┤
│ startAutoRefresh()                  │
│ ├─ Créer ScheduledExecutorService   │
│ ├─ Planifier refreshDataFromDatabase()
│ └─ Log: ✅ Auto-refresh démarré    │
├─────────────────────────────────────┤
│ Boucle (toutes les 5s)              │
│ ├─ Thread 1: refreshDataFromDatabase()
│ │  ├─ Requête BD (getAllBudgets...)
│ │  └─ Retour ObservableList         │
│ ├─ Platform.runLater() → UI Thread  │
│ ├─ baseList.setAll(newData)         │
│ ├─ updateKpis()                     │
│ ├─ updateCharts()                   │
│ ├─ applyPredicate()                 │
│ └─ statusLabel.setText(...)         │
└─────────────────────────────────────┘

Thread-safety :
- Données = Thread du scheduler (async)
- UI Update = UI/FX Thread (Platform.runLater)
- ✅ Pas de race conditions

📈 PERFORMANCE & CONSIDÉRATIONS

✅ Points forts :
├─ Transparent pour l'utilisateur
├─ Ne bloque pas l'UI
├─ Gère automatiquement les changements
├─ Propre (baseList.setAll())
├─ Logs pour debug
└─ Thread-safe

⚠️ Considérations :
├─ 1 requête BD toutes les 5s par module
├─ 3 modules = ~36 requêtes/minute
├─ Si BD distante → vérifier latence
├─ Augmenter intervalle si surcharge
└─ Cache possible pour optimiser

💡 Optimisations futures :
├─ Delta sync (charger seulement changements)
├─ WebSockets pour temps réel
├─ Augmenter intervalle si inactif
├─ Cache de dernière requête
└─ Notification utilisateur des changements
