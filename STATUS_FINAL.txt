💡 COMMENT ÇA MARCHE

Cycle de rafraîchissement (toutes les 5 secondes) :

1️⃣  ScheduledExecutorService déclenche
2️⃣  Thread background : Requête BD (SELECT * FROM ...)
3️⃣  Données reçues : List<Sponsor/Budget/Depense>
4️⃣  Platform.runLater() : Retour au FX Thread (thread-safe)
5️⃣  baseList.setAll(newData) : Mise à jour ObservableList
6️⃣  Observable notifie les listeners
7️⃣  renderCards() : Réaffichage des cartes
8️⃣  updateKpis() : Statistiques à jour
9️⃣  updateCharts() : Graphiques régenérés
🔟 applyPredicate() : Filtres réappliqués
1️⃣1️⃣ UI mise à jour ✅

📈 ARCHITECTURE TECHNIQUE

Design Pattern : Reactive Programming + Scheduled Tasks

Components :
├─ ScheduledExecutorService : Gère le timing (5s)
├─ Thread Pool : Exécute les requêtes BD
├─ Platform.runLater() : Thread-safety
├─ ObservableList : Déclenche les mises à jour
└─ Listeners : Réagissent aux changements

Thread Safety :
├─ BD Thread (async) : Récupère données
└─ FX Thread : Met à jour UI
    └─ Pas de race conditions
    └─ Pas de deadlocks
    └─ UI reste fluide

🔒 SÉCURITÉ & PERFORMANCE

Considérations :
├─ 1 requête/5s par module = 36 requêtes/minute
├─ 3 modules = ~108 requêtes/minute
├─ BD distante = vérifier latence réseau
├─ Augmenter intervalle si surcharge serveur
└─ Cache possible pour optimiser

Pas de problème avec :
├─ Injection SQL (requêtes via ORM)
├─ Race conditions (Platform.runLater)
├─ Memory leaks (scheduler géré proprement)
├─ UI freeze (thread-safe)
└─ Erreurs de base de données (try-catch)
